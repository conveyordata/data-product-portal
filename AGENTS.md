# AI Agent Guidelines for Data Product Portal

Welcome! These guidelines will help you navigate and contribute effectively to the Data Product Portal codebase. Keep this context concise to minimize token usage while retaining critical architectural boundaries.

## Project Overview
The Data Product Portal manages data products and their output ports. It consists of a multi-tier architecture with a FastAPI backend, a React frontend, a CLI, an MCP server, and comprehensive documentation.

## Tech Stack
*   **Backend**: Python ≥3.13, FastAPI, SQLAlchemy (async), Alembic, Casbin (RBAC), pgvector (embeddings), Pydantic v2, FastMCP.
*   **Backend tooling**: Poetry (package management), Ruff (linting/formatting), mypy (type checking), pytest (testing).
*   **Frontend**: React 19, TypeScript, Vite, Redux Toolkit + RTK Query, Ant Design, Biome (linting/formatting), i18next (localization), Vitest (testing).
*   **CLI**: Go (`cli/golang/`), Python (`cli/python/`).
*   **MCP**: FastMCP server exposing portal resources to AI agents.
*   **Docs**: Docusaurus.
*   **Tooling**: `Taskfile.yml` (release automation), Docker, Helm.

## Directory Structure
*   `backend/app/`: Python API. Follows Service-Router-Model-Schema pattern.
*   `frontend/`: React SPA.
*   `cli/`: Command-line tools (Go & Python).
*   `mcp/`: MCP server for AI agent access to the portal.
*   `demo/`: Demo setups.
*   `docs/`: Docusaurus documentation and Architecture Decision Records (ADRs in `docs/adr/`).
*   `integrations/`: Platform-specific integration logic (Terraform, BitOL, data quality).
*   `helm/`: Kubernetes deployment charts.
*   `nginx/`: Reverse proxy configuration.
*   `scripts/`: Build and utility scripts.

## Backend Guidelines
*   **Architecture**: Features are modularly isolated in `backend/app/<feature>/` (e.g., `data_products`). Adhere strictly to the separation of `router.py`, `service.py`, `model.py`, `schema_request.py`, and `schema_response.py`.
*   **Schemas**: Strictly use Pydantic v2 for validation. Differentiate between request (`schema_request.py`) and response (`schema_response.py`) schemas. Some modules also have a shared `schema.py`.
*   **Authorization**: The system uses Casbin for fine-grained RBAC. Always apply `Authorization.enforce()` dependencies on new routers and endpoints.
*   **Database**: Use SQLAlchemy async models and Alembic for migrations (`backend/app/database/alembic/versions/`).
*   **Testing**: Run `cd backend && poetry run poe test` (or `poetry run pytest -v tests/`). Tests live in `backend/tests/`, organized to mirror `backend/app/`.
*   **Linting/Formatting**: Run `poetry run poe lint` and `poetry run poe format`. Ruff is the primary tool; mypy enforces types.

## Frontend Guidelines
*   **API Client**: `frontend/src/store/api/` contains auto-generated RTK Query service files from the OpenAPI spec. **Never edit these files manually** — they are regenerated by `npm run generate-api`.
*   **Types**: Ensure TypeScript interfaces strictly match the backend's OpenAPI schemas.
*   **UI Components**: Use Ant Design components. When needing to make new components follow existing component patterns in `frontend/src/components/`.
*   **Localization**: Use `i18next` for all user-facing strings. No hardcoded text. Run `npm run extract-translations` after adding new strings.
*   **Code Quality**: Run `npm run check` (Biome) before committing. The pre-commit hook runs `npm run check:fix` automatically.
*   **Testing**: Run `npm test` (Vitest). Tests live in `frontend/src/tests/`.

## Workflow & Automation
*   **Task Runner**: `Taskfile.yml` is primarily for release automation. Use `poetry run poe <task>` for backend dev tasks and `npm run <script>` for frontend dev tasks.
*   **OpenAPI Updates**: If you modify backend routes or schemas, run `task update:open-api-spec` to regenerate the OpenAPI specification and the frontend RTK Query client before completing the task.
*   **Pre-commit**: Code must pass `pre-commit` hooks (Ruff, mypy, Biome, gitleaks, OpenAPI spec check).
*   **Documentation**: When adding major features or making architectural changes, consider if a new ADR (`docs/adr/`) is needed.

## General Rules for Agents
*   **Analyze First**: Use search tools to find and replicate existing patterns before inventing new ones.
*   **Surgical Changes**: Modify only what is necessary. Avoid unsolicited refactoring.
*   **Validation**: Always verify your changes — run `poetry run poe test` for backend, `npm test` for frontend.
