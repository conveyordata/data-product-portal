Three Approaches: Cost Tracking Architecture

How should cost data be structured and served?

A

Sub-module (recommended) — mirror the data quality pattern

Create output_ports/cost/ as a self-contained sub-module with its own model, router, service, and schemas — exactly like output_ports/data_quality/ today.

  • POST endpoint to push cost records (external billing systems)
  • GET endpoint returns full cost history for a given output port
  • Separate GET endpoint for upstream cost exposure (server computes proportional share)
  • Frontend: cost history chart + upstream panel on output port page sidebar; cost breakdown per output port on data product page

Pros

  • Follows established codebase pattern
  • Clean isolation — cost logic lives in one place
  • Easy to extend later (new categories, more aggregations)

Cons

  • More files to create
B

Embed cost in the output port model

Add cost columns directly to the datasets table. No separate model or sub-module — cost fields are part of the core output port.

  • Single PATCH endpoint to update costs on an output port
  • No history table — just current values
  • Upstream exposure computed inline in the existing output port service

Pros

  • Fewer files, simpler initial implementation

Cons

  • Contradicts the historical-series requirement
  • Pollutes the core dataset model
  • Harder to evolve later
C

Top-level cost module (separate from output ports)

Create backend/app/costs/ as a first-class top-level feature, not nested under output ports. Cost records reference output ports by ID but live in their own domain.

  • Own router at /v2/costs/
  • Can later expand to cost at data product level, domain level, etc.
  • Service handles all aggregation logic independently

Pros

  • Most extensible — easy to add cost tracking to other entities

Cons

  • Breaks the existing pattern of sub-modules per feature
  • Overkill for the current scope
